## Feature branch naming: dashes instead of slashes

- **What:** kickoff prompt specifies feature branches as `ship-wave1/cr-phase-1` (slash-nested under the integration branch name).
- **Why deferred:** git refuses to create a branch nested under an existing branch name (`ship-wave1/scaffold` collides with existing ref `ship-wave1` — refs/heads is a filesystem-like namespace, a name can't be both a file and a directory).
- **Default taken:** feature branches use dashes instead: `ship-wave1-scaffold`, `ship-wave1-cr-phase-1`, `ship-wave1-cr-phase-2`, etc. Same intent (one branch per package, merges into `ship-wave1`), trivially reversible (rename before pushing if the Captain prefers slashes with a differently-named integration branch, e.g. `wave1` instead of `ship-wave1`).

## Package 3 (Chart Room phase 3, editor) plan review — decisions taken by First Officer

- **Round-trip strategy — approved as designed.** Block-level diff-and-splice against the original raw text (mirrors phase 1's `fix-links.ts` byte-offset discipline), with a strict-allowlist opaque-passthrough node for anything Milkdown's presets don't understand (`html`, `remark-directive` blocks, and defensively any unrecognized node type). The Team Lead's research confirmed, concretely, that whole-document Milkdown/`remark-stringify` serialization would (a) rewrite bullet/heading style on essentially every real pre-existing doc and (b) actively corrupt `:::llm`/`:::human`/`:::ask-me` directive blocks since Milkdown's default presets can't parse `remark-directive` syntax at all — this is a real, confirmed finding, not a hypothetical, and the block-diff design is the right response to it. Approved.
- **New dependencies**: `@milkdown/kit` (^7.21.2, not `@milkdown/crepe` — the composable-primitives package is the right pick given the customization this phase needs), `@milkdown/react` (^7.21.2), `fuse.js` (^7.4.2) as new `chartroom-ui` runtime deps; `remark-frontmatter` (^5.0.0) newly added to `chartroom-ui` (already a phase-1 dep, same version). Approved — Milkdown is spec-named (§7), the rest is small/MIT/zero-network per the established approval pattern. `cmdk` and a dedicated diffing library both correctly rejected as unnecessary per the project's "don't add a dependency for ~50-100 lines" precedent.
- **Cross-package boundary: exporting phase-1's offset-walking primitives for `chartroom-ui` to import.** Approved — this is the one place the plan deviates from phase 2's "duplicate rather than touch phase 1/2 files" precedent, but for a good, narrow reason: byte-offset extraction across 7+ node types is correctness-critical for the round-trip guarantee, and phase 1 already had one documented empirical false-start getting this logic right (phase 1 plan §10 risk #3) — reusing the tested implementation is safer than re-deriving it under phase-3 time pressure. - **`fix-links.ts` behavior extension for image-href repair.** Approved — the one actual behavior change (not just an export addition) to existing phase-1 code, needed to make "pasted image self-heals after `git mv`" literally true rather than true by accident. ## Package 3 (Chart Room phase 3, editor) — Developer-stage findings and deviations from the approved plan

Implemented on `ship-wave1-cr-phase-3`. The plan's own §1.7 honesty flag ("Milkdown's exact API surface... could not be confirmed... a real, named gap") was taken seriously: a first spike installed `@milkdown/kit`+`@milkdown/react` and read the installed `.d.ts`/`.js` files directly before writing any real code. Findings below are what actually changed versus the approved plan, plus a build-graph problem the plan's own approved cross-package-export decision (item 3 above) turned out to require fixing.

- **Milkdown API spike: mostly confirmed, one real design-invalidating conflict found.** `useEditor`/`<Milkdown/>`/`MilkdownProvider`, `Editor.make()`/`rootCtx`/`defaultValueCtx`, `listener`/`listenerCtx`, `$node`, `$remark`, `parserCtx`/`serializerCtx`/`schemaCtx`, `commonmark`/`gfm` preset arrays all matched the plan's assumed shapes exactly. One real divergence: `@milkdown/preset-commonmark` already registers its own node schema for mdast type `'html'` — but as an **inline** atom (`group: 'inline'`, for a `<span>` mid-paragraph), not a block-level one. The plan's literal §3.1 step 4 design ("build the Milkdown document by concatenating... a custom node for each opaque block") sidesteps this cleanly by never feeding whole raw text (with directive/HTML syntax in it) through Milkdown's own markdown-string parser at all — prose blocks are parsed via `parserCtx` on their own isolated text; opaque blocks are constructed directly as `chartroomOpaqueBlock` node instances. An earlier, since-abandoned implementation attempt (feeding the whole body through Milkdown's parser with `$remark(remarkDirective)` wired in) hit this conflict directly; reverting to the plan's literal per-block-construction design resolved it. Documented in `opaqueNode.ts`'s file header.
- **A real circular-workspace-dependency risk in the approved cross-package export (item 3 above), found and fixed.** Making `chartroom-ui` depend on `chartroom` (for the `./markdown` subpath export) while `chartroom` already depended on `chartroom-ui` (devDependency, purely to order the UI-bundle-copy step in its own build script) created a genuine cycle — confirmed via `turbo run build --dry` before making any change, which printed `Cyclic dependency detected: chartroom-ui#build -> chartroom#build -> chartroom-ui#build`. Fixed by splitting `chartroom`'s single `build` script (tsc + copy) into `build` (tsc only) and a new `build:ui-bundle` script/turbo task explicitly declared (via turbo.json's `<package>#<task>` override syntax) to depend on both packages' own `build` tasks — re-verified acyclic via the same `--dry` flag afterward. Root `package.json`'s `build` script updated to run both (`turbo run build build:ui-bundle`) so a single `pnpm build` still produces the complete self-contained bundle in one pass. A small, mechanical, necessary consequence of the approved decision, not a design change — flagged here since it touches files outside the phase's named file list (`turbo.json`, root `package.json`, `chartroom/package.json`'s scripts).
- **A confirmed (not hypothetical) browser-runtime break, found and fixed.** The Ctrl+K link picker's `insertLink.ts` originally imported `computeExpectedHref` from the new `chartroom/link-paths` export (an additional, not-yet-mentioned-above cross-package export this session added, since removed again — see below). `link-paths.ts` uses `node:path`'s `posix` namespace internally. An actual `npx vite build` of `chartroom-ui` was run and its output inspected directly: Vite/rolldown externalizes `node:path` for a browser bundle as an **empty stub object**, not a working polyfill — calling `computeExpectedHref` at runtime (i.e. on every Ctrl+K link insertion) would throw `TypeError: Cannot read properties of undefined (reading 'dirname')` in an actual browser with no Node.js present. Fixed by reimplementing `computeExpectedHref`/`normalizeHref` as pure string operations with zero `node:path` dependency (`chartroom-ui/src/editor/relativeHref.ts`), cross-checked against the real `chartroom` implementation across 15 path-pair cases (all match exactly), and removing the now-unused `./link-paths` export from `chartroom`'s `package.json` again. The `./markdown` export (item 3 above) is unaffected — `segmentBlocks.ts` only ever imports `AstNode` as a `import type`, which is fully erased at build time and never reaches the browser bundle, so it carries no equivalent risk. Worth a general note for future phases: **any cross-package import that will execute inside `chartroom-ui`'s browser bundle needs to be verified against a real `vite build`'s output, not just `tsc`** — type-checking success does not catch a Node-builtin dependency silently breaking at runtime.
- **`fix-links.ts`'s image-href-repair mechanism (item 4 above) needed a second mechanism beyond the plan's literal content-hash description, to actually close the gap it claims to close.** The plan's own §6.3 describes "look up its resolved absolute path's content hash against `index.assets`" — but on closer implementation, this has a real bootstrapping problem: `indexer.ts::collectAssets` only discovers/hashes an image by successfully resolving it from a doc's *currently-written* href; once a doc moves and its image href becomes stale (the literal acceptance scenario), that asset is *not* rediscoverable via any doc's resolvable reference, so it's never hashed on the next rebuild, and `index.assets` has no entry for it to look up. Implemented `computeImageFixes` with the content-hash mechanism as described **plus** a second, deterministic fallback: since this project's own upload endpoint always writes assets to `assets/<hosting-doc-id>/<timestamp>.png` (a folder keyed by the doc's own `id`, unaffected by the doc's own path), a broken href whose basename matches a file actually present under the current doc's own `assets/<id>/` folder is repaired directly from that naming convention, no hash lookup needed. Both mechanisms are implemented, tested (6 new `fix-links.test.ts` cases), and exercised end-to-end by the new acceptance script. Documented in `fix-links.ts`'s own doc comment for `computeImageFixes`.
- **`acceptance/editor-round-trip.mjs` scope, narrowed deliberately.** The plan's §9.4 describes this script also headlessly re-exercising `roundTrip.ts`'s no-op round-trip claim. `chartroom-ui`'s build output (`vite build`) is a bundled SPA, not a set of Node-importable ES modules the way `chartroom`'s own `tsc`-built `dist/` is, so a plain `.mjs` script can't reuse `roundTrip.ts` the way phase-1/phase-2's acceptance scripts reuse `chartroom`'s own CLI/daemon modules — doing so would require adding `jsdom` as a new `chartroom` dependency (not on the approved list) purely to duplicate a suite that already passes (`roundTrip.test.ts`, 40/40). The script instead covers only the image-paste-then-`git mv`-self-heal half of the acceptance line end-to-end (which lives entirely in `chartroom`'s own daemon+CLI code), with the no-op-round-trip half's coverage noted explicitly in the script's header and console output rather than silently dropped.
- **Real-browser QA — attempted, same outcome as phase 2.** Per the standing resolution above, no automated Playwright was added. A live `claude-in-chrome` click-through was not attempted this session (out of scope for the Developer stage per the phase-2 precedent, reserved for the Reviewer pass) — flagging again, now with a larger interactive surface (Ctrl+K, paste events, a mounted third-party rich-text editor) than phase 2 had, so the case for an eventual real-browser check is stronger. The structural proxy used instead: real `tsc`, `vite build`, and all three `chartroom` acceptance scripts (including the new `editor-round-trip.mjs`, itself a real Fastify `.inject()` exercise of the actual route code, not a mock) all pass.
- Nothing else needing `REMOVALS.md` beyond the throwaway spike test file already logged there (`packages/chartroom-ui/test/editor/_spike.test.ts`, left deliberately uncommitted). `team-tasks/` untouched.
