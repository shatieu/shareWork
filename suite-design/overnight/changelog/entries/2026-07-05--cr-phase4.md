# Chart Room Phase 4 — Interactive Blocks + Human-Action Inbox

Delivered per `ChartRoom_Spec.md` §8 build-order item 4:

- **`:::ask-me` questions**, reusing the real `.claude/skills/ask-human/SCHEMA.md` type vocabulary
  (`single-select`, `multi-select`, `text`, `yesno`, `rating`, `ranking`, `compare`), with the spec's
  own example (`type="choice"`) treated as a resolved backward-compatible alias rather than a
  fictitious eighth type. Seven fresh React widgets mirror the existing ask-human skill's browser UX.
  Answers are spliced into the doc in place — `answered="true"` on the directive plus a
  human-readable `> **Answer** (date, author): ...` blockquote — via a new, narrow server-side
  `PATCH .../ask-me` endpoint that re-reads the file fresh from disk and touches only that one
  directive's byte span.
- **GFM checkbox write-back** and **`:::actions`** (one directive per human-action item, symmetric
  with `:::ask-me`'s shape): a new `PATCH .../checkbox` endpoint does a true single-character splice.
  Both write paths are architecturally separate from phase 3's Milkdown editor — `:::ask-me`/
  `:::actions` blocks stay non-editable inside Milkdown by design; these are read-only-viewer
  interactions with their own server-side splice logic, not a reuse of the editor's save path.
- **Cross-repo human-action inbox** (`GET /api/inbox` + a new `#/inbox` page): aggregates every
  unanswered ask-me question and unchecked action across all registered repos, computed once per
  index rebuild (no re-parsing on the read path).
- A new cross-package runtime export (`chartroom/interactive-blocks`, actual executable code this
  time, not just a type as in phase 3) — verified dependency-free of Node builtins via a real `vite
  build` inspection before any UI was built on top of it, per the mandatory checkpoint phase 3's own
  postmortem required.

**One real bug found by the adversarial Reviewer and fixed before merge:** every interactive checkbox
rendered permanently unclickable in a real browser. `mdast-util-to-hast` (already bundled) hard-codes
`disabled: true` on every GFM task-list checkbox's hast node; `Checkbox.tsx` was passing that through
verbatim instead of forcing it off for the interactive case. All 267 unit tests, four acceptance
scripts, `tsc`, and `eslint` missed this because `fireEvent.click` in jsdom/RTL bypasses a disabled
input's native click-suppression — a real browser wouldn't. Fixed by unconditionally forcing
`disabled={false}` whenever a checkbox resolves to a real, addressable `CheckboxRef`, with a new
regression test that supplies `disabled={true}` the way real GFM parsing always does. Two smaller
issues found in the same pass were fixed alongside it: an `:::actions` directive with more than one
checkbox was only checking the first one's state (now requires all checkboxes checked); two `:::ask-me`
blocks sharing an `id` would silently answer the wrong one (now rejected with 409).

Verified post-fix: `chartroom` 136/136 tests, `chartroom-ui` 135/135 tests, `tsc`/`eslint` clean in
both, all four `chartroom` acceptance scripts (phases 1-4) pass, a real daemon boot across two scratch
repos with a mix of answered/unanswered/checked/unchecked fixtures confirmed `GET /api/inbox` returns
exactly the correct pending subset.

Two known limitations accepted for v1, logged in `DECISIONS-NEEDED.md` rather than silently omitted:
no in-app way to correct an already-answered ask-me question (only a hand-edit of the raw file), and
the optimistic-concurrency guard on checkbox/ask-me writes doesn't protect against a concurrent
Milkdown full-file save racing one of them (consistent with phase 3's already-accepted
last-write-wins posture for a single-local-user tool).

**Fourth consecutive phase without real-browser QA** — the Chrome extension was unavailable again
this session (attempted twice: phase 2 and phase 4). Flagging this prominently rather than deferring
silently again; notably, this phase's real bug was still caught, by careful adversarial reasoning
about jsdom-vs-real-browser semantics rather than an actual browser, which is a useful but partial
substitute.
