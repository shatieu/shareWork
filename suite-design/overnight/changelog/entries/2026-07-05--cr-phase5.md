# Chart Room Phase 5 — Agent Surface Polish (final phase)

Delivered per `ChartRoom_Spec.md` §8 build-order item 5, completing all five Chart Room phases:

- **MCP server**: one `McpServer` instance, five tools (`resolve`, `read_doc`, `search`,
  `list_unanswered_questions`, `answer_status`) as thin wrappers over phases 1/4's existing
  resolver/index/interactive-blocks logic — zero new detection/extraction logic. Two transports:
  `StdioServerTransport` for a new `chartroom mcp` CLI command (cwd-scoped, single-repo), and
  `StreamableHTTPServerTransport` mounted per registered repo into the existing daemon at
  `/api/repos/:repoId/mcp`, in stateless mode (a fresh server+transport per HTTP request, matching
  the SDK's own shipped example — an early attempt at one long-lived instance per repo failed on a
  second request and was corrected).
- **`chart-room` skill** (packaged template + `chartroom install-skill`): teaches an agent to resolve
  dead paths via the index/CLI/MCP, write links in id format, use `:::llm`/`:::human` conventions
  (including explicitly skipping `:::human` bodies for token savings), and post/check `:::ask-me`
  questions — matching this repo's own real `ask-human` skill's structural convention exactly.
- **Agent hook, corrected mid-implementation**: the plan assumed a `PostToolUse` hook fires on a
  failed Read; the Developer found, by fetching Claude Code's own hooks documentation, that
  `PostToolUse` fires only on *success* — a separate `PostToolUseFailure` event exists for failures.
  Corrected before merge. The hook's detection doesn't depend on guessing the failure payload's exact
  field shape (undocumented) — it gates on whether `chartroom resolve` finds a genuinely different
  path than the one that failed, a more robust design than the plan's original string-match idea.
  Independently re-verified against the live docs by the adversarial Reviewer.
- **CLAUDE.md template line** and **`chartroom llms-txt`** (index-derived, repo-relative paths,
  matching the llms.txt convention's spirit for a local repo rather than a hosted site).

Verified: 177/177 tests, `tsc`/`eslint` clean, all five acceptance scripts (phases 1-5) pass. The
Reviewer independently drove all five MCP tools against real edge cases (tombstoned id, not-found id,
ambiguous duplicate `:::ask-me` id across two docs) via a real spawned `chartroom mcp` subprocess and
a real MCP `Client`, and stress-tested the HTTP transport with 5 sequential + 2 concurrent calls
beyond what the committed test suite covered. `packages/chartroom-ui` has zero changes this phase.

**Dogfooding Chart Room onto `shareWork` itself — deliberately not done.** Every install/test
operation ran only against disposable scratch repos; this repo's own `.claude/` directory and docs
are untouched, no `.docs/index.json` exists at the root. This was a real, one-way mutation decision
correctly deferred to the Captain rather than taken as a default action.

**Two honest, mission-level gaps surface at the end of this final phase, not fixed by it:**
1. Spec §9's DoD line ("ask-me/checklist/llm blocks work end-to-end with a real Claude Code session")
   has never been verified with an actual live session across any of the five phases — every
   acceptance proof this whole project produced is an automated script, never a real agent choosing
   to use these tools unprompted. Structurally unautomatable, not a shortfall of this
   implementation, but formally unproven.
2. Staleness-rule growth (`ttl_days`/`sources:` freshness gates, orphan detection) — narratively
   promised in spec §6, traced by phase 5's own Team Lead through every literal Build Order
   acceptance line, and found to have never actually been assigned to any phase's real acceptance
   criteria. A genuine spec gap, not a phase-2 or phase-5 shortfall — flagged for the Captain to
   decide (accept as dropped for v1, or assign to a future pass).

Both logged in `DECISIONS-NEEDED.md` and carried into the morning report rather than silently
resolved or dropped.
